# گزارش نقاط ضعف پروژه SMM Turk

این سند بر اساس بررسی فولدرها، فایل‌های PHP، امنیت و معماری پروژه تهیه شده است.

---

## ۱. امنیت (Security)

### ۱.۱ کلید Session و CSRF (SECRET_KEY)
- در `config.example.php` مقدار `SECRET_KEY` با `bin2hex(random_bytes(16))` در هر اجرا تولید می‌شود.
- در production اگر این مقدار ثابت نباشد، با هر restart سرور همه sessionها باطل می‌شوند و کاربران از حساب خارج می‌شوند.
- **توصیه:** در production حتماً یک مقدار ثابت ۳۲ کاراکتر hex (مثلاً از `openssl random -hex 16`) در `config.php` یا env قرار دهید و از تولید تصادفی در runtime خودداری کنید.

### ۱.۲ کوکی Session
- برای `session_start()` هیچ `session_set_cookie_params()` با گزینه‌های `HttpOnly`, `Secure`, `SameSite` تنظیم نشده است.
- در نتیجه احتمال سوءاستفاده از XSS برای دزدیدن session یا حملات session fixation بیشتر است.
- **توصیه:** قبل از `session_start()` در `Auth.php` یا `init.php`:
  - `session_set_cookie_params(['httponly' => true, 'secure' => true, 'samesite' => 'Lax'])` (در HTTPS حتماً `secure => true`).

### ۱.۳ Content-Security-Policy (CSP)
- هیچ هدر CSP تنظیم نشده است.
- **توصیه:** برای کاهش خطر XSS و تزریق اسکریپت، یک CSP مناسب (حتی گزارش‌گیر در ابتدا) اضافه شود.

### ۱.۴ API Key در URL
- در `api/v2.php` کلید API از هر دو `$_GET['key']` و `$_POST['key']` خوانده می‌شود.
- اگر کاربر key را در URL قرار دهد، ممکن است در لاگ سرور، referer یا تاریخچه مرورگر ذخیره شود.
- **توصیه:** فقط از `POST` یا هدر (مثلاً `X-API-Key`) برای key استفاده شود و در مستندات API این موضوع قید شود.

### ۱.۵ Rate Limit لاگین
- محدودیت لاگین بر اساس IP است (۵ تلاش در ۱۵ دقیقه). در محیط‌های اشتراکی (شرکت، VPN، NAT) چند کاربر ممکن است یک IP داشته باشند و یک نفر باعث قفل شدن بقیه شود.
- می‌توان در کنار IP، محدودیت بر اساس شناسه (مثلاً ایمیل) هم در نظر گرفت (اختیاری).

### ۱.۶ نمایش خطای دیتابیس
- در `Database.php` در صورت خطای اتصال با `die(json_encode(['error' => '...' . $e->getMessage()]))` پیام دقیق PDO به کاربر نشان داده می‌شود.
- در production ممکن است مسیر دیتابیس، نام کاربر و غیره لو برود.
- **توصیه:** در production فقط یک پیام کلی به کاربر نشان داده شود و جزئیات در لاگ نوشته شود.

## ۲. پیکربندی و تنظیمات

### ۲.۱ مقدار نمونه کیف پول در config.example
- `CRYPTO_WALLET_ADDRESS` در `config.example.php` با یک آدرس واقعی پر شده است. بهتر است با placeholder مثل `'YOUR_ETH_OR_USDT_ADDRESS'` جایگزین شود تا به اشتباه در repo باقی نماند.

### ۲.۲ حالت Debug در production
- اگر در production مقدار `SMM_DEBUG` یا env درست تنظیم نشود، `display_errors` ممکن است روشن بماند و خطاهای PHP و مسیر فایل‌ها به کاربر نشان داده شود.
- **توصیه:** در `config.php` پیش‌فرض را `SMM_DEBUG = false` قرار دهید و فقط با تنظیم صریح env یا constant آن را روشن کنید.

### ۲.۳ ناسازگاری محتوای لندینگ با واقعیت
- در صفحه اصلی و فایل‌های زبان (مثلاً `secure_payment_desc`) روش‌های پرداخت مثل کارت، PayPal، Payeer و غیره ذکر شده، در حالی که در کانفیگ فقط `CRYPTO_WALLET_ADDRESS` برای واریز وجود دارد.
- **توصیه:** متن‌های لندینگ و FAQ را با روش پرداخت واقعی (فعلاً فقط کریپتو) هماهنگ کنید تا اعتماد کاربر خدشه‌دار نشود.

## ۳. معماری و کیفیت کد

### ۳.۱ CSS و JS پراکنده
- بیشتر استایل‌ها در `layouts/header.php` و `home.php` به صورت inline هستند؛ فولدرهای `assets/css` و `assets/js` عملاً خالی (.gitkeep) هستند.
- نگهداری، theme و cache کردن سخت‌تر می‌شود.
- **توصیه:** در بلندمدت استایل‌های مشترک را در فایل‌های CSS جدا و اسکریپت‌های مشترک را در JS جدا قرار دهید.

### ۳.۲ خروجی ترجمه و XSS
- در `home.php` برای برخی رشته‌ها از `__('...')` بدون `h()` استفاده شده (مثلاً `hero_desc_1` با تگ `<strong>`). اگر در آینده ترجمه‌ها توسط کاربر یا از منبع غیرمطمئن بیاید، خطر XSS وجود دارد.
- **توصیه:** برای همه خروجی‌های متنی به HTML از `h()` استفاده شود؛ در صورت نیاز به HTML مجاز، از یک allowlist تگ محدود استفاده شود.

### ۳.۳ نسخه‌گذاری API
- مسیر API به صورت `/api/v2` است ولی versioning واقعی (مثلاً در هدر یا query) برای سازگاری با نسخه‌های بعدی تعریف نشده است.
- **توصیه:** برای تغییرات ناسازگار بعدی یک مسیر مثل `/api/v3` یا پارامتر version در نظر بگیرید تا کلاینتهای قدیمی خراب نشوند.

## ۴. عملکرد (Performance)

### ۴.۱ بارگذاری لیست سرویس‌ها
- در `index.php` وقتی دسته‌بندی انتخاب نشده، تا ۱۵۰۰ سرویس یکجا از دیتابیس خوانده و در صفحه نمایش داده می‌شود.
- در `services.php` حد ۵۰۰ رکورد با LIMIT وجود دارد؛ در index با ۱۵۰۰ ممکن است صفحه اول برای کاربران با اینترنت ضعیف سنگین شود.
- **توصیه:** pagination یا lazy load برای لیست سرویس‌ها در صفحه سفارش، یا محدود کردن تعداد با اسکرول بی‌نهایت.

### ۴.۲ کوئری جستجو در tickets
- در `tickets.php` جستجو با `LIKE '%...%'` انجام می‌شود. روی جدول بزرگ بدون ایندکس مناسب ممکن است کند شود.
- در صورت رشد جدول `tickets`، ایندکس روی `subject` و `order_id` یا full-text را در نظر بگیرید.

---

## ۵. فایل‌ها و آپلود

### ۵.۱ آپلود تیکت
- در `tickets.php` پسوند و MIME برای فایل‌های ضمیمه چک می‌شود و فایل در پوشه جدا ذخیره می‌شود؛ `uploads/.htaccess` اجرای PHP را مسدود کرده است — از نظر امنیتی قابل قبول است.
- نام فایل با `preg_replace` ساده‌سازی می‌شود؛ اگر در آینده فایل با نام خیلی طولانی آپلود شود، محدودیت طول نام در دیتابیس یا filesystem را در نظر بگیرید.

## ۶. دیتابیس و Migration

### ۶.۱ ادمین پیش‌فرض
- در `install.sql` کاربر ادمین پیش‌فرض ساخته نمی‌شود که از نظر امنیتی خوب است؛ ولی باید بعد از نصب اولین بار یک ادمین به صورت دستی (یا از طریق اسکریپت امن) ایجاد شود.
- **توصیه:** در مستندات نصب (مثلاً CHECKLIST-DEPLOY یا README) مرحله «ساخت اولین کاربر ادمین» به وضوح ذکر شود.

## ۷. خلاصه اولویت‌ها

| اولویت | مورد | اقدام پیشنهادی |
|--------|------|------------------|
| بالا | SECRET_KEY ثابت در production | در config.php مقدار ثابت قرار دهید |
| بالا | Session cookie (HttpOnly, Secure, SameSite) | session_set_cookie_params قبل از session_start |
| بالا | عدم نمایش جزئیات خطای DB به کاربر | در production فقط لاگ + پیام کلی |
| متوسط | API key فقط در body/header | مستندات + تغییر v2 یا در نسخه بعد |
| متوسط | هماهنگی متن لندینگ با روش پرداخت | به‌روزرسانی متن‌ها و FAQ |
| متوسط | CSP | اضافه کردن هدر CSP (حتی report-only) |
| پایین | جداسازی CSS/JS | refactor تدریجی |
| پایین | Pagination/lazy load سرویس‌ها | برای بهبود تجربه کاربری |
